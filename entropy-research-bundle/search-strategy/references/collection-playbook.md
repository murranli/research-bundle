# 采集打法手册（strategy 阶段参考）

> 迁移自 v1.0「采集指南」的策略部分（渠道目录见 `channel-map.md`）。

## STEP A · 概念抽取与同义词泛化
- 从子问题抽核心概念：实体名词（公司/产品/品牌）、商业指标、产品特性。
- 基于"泛商业/产品"语境做同义词与术语泛化：覆盖中英文、口语与术语、旧称与新称、缩写与全称。
- **泛化纪律**：必须是该领域真实通用的术语/别称，不为凑数硬塞沾边近义词，宁少勿滥。

## STEP B · 框定时空范围
- **时间基准**：算"近N年/最新/今年/近期"时，以 manifest 注入的当前日期为基准回推，不要凭记忆假设年份。
- 先读 manifest `original_request`/`scope` 与 intent 产物 `constraints` 的 time_range、geography。
- 若为空：自主判断最有利采集的时空范围并说明理由。
- **默认新鲜度倾向**：上游未指定且子问题不指向特定历史年份时，默认倾向【当前日期前约 1.5 年内】，与下游审计的新鲜度门槛对齐；这是倾向非硬限，权威信源或确需更早数据时不受限。

## STEP C · 话题类型判定（A/B/C，决定渠道与检索式倾向）
读子问题的坐标与 `object_dimension`（及 purpose）判定：
- **A 客观事实/数据**：维度含 财务与资本运作 / 市场基本面 / 技术壁垒 / 宏观环境，或求具体数值、参数、政策条款。
- **B 产品/运营/策略**：维度含 营销运营 / 产品能力 / 商业模式 / 组织结构，问"怎么设计/运营/什么打法机制"。
- **C 口碑/体验/情绪**：维度含 客群体验，问"用户怎么评价、真实体验、满意度、吐槽"。

下游 `retrieval-exec` 的两层筛选用同一套 A/B/C，确保上下游一致。

## STEP D · 检索策略选择
为每个子问题指定一种或多种（可组合）：中性检索、宽搜、精搜、反向/证伪搜索、竞品并列搜索、时间限定搜索、雪球式搜索。
- 策略与子问题性质匹配（对比类→竞品并列；找最新→时间限定）。
- **必含证伪策略**：每个子问题必须含一种"反向/证伪搜索"，直接对应其 `falsification`。

## STEP E · 检索式构建（每子问题 2–4 组）
**硬性要求**：
- 必须用操作符提高信噪比，至少含——精准匹配 “”、逻辑或 OR、剔噪 -（减号）中的至少两种组合，不能裸关键词堆叠。
- **引号纪律**：精准匹配一律用中文全角引号 “”，**严禁英文双引号 "**——检索式处于输出 JSON 的字符串值内部，英文双引号会破坏 JSON 致下游解析失败。搜索引擎对全角引号同样精准匹配，不损失效果。
- **时间注入**：采用默认新鲜度倾向时，至少在「精搜+剔噪」或「时间限定」一组把年份写进检索式（如 2025 2026），让源头偏新；「宽搜」一组保持克制不强加时间以保召回。
- **宽窄搭配**：至少 1 组宽搜（保召回）+ 1 组精搜+剔噪（“”锁定 + - 剔噪，保精度）+ 1 组证伪式（搜反例/质疑/翻车/差评）。
- 单组减号剔除项 ≤ 3 个，避免约束过狠零结果。

> 注：检索式是给"关键词型"渠道（exa 语义搜索、社媒搜索）的；语义搜索可用更自然的长句，关键词搜索才需要操作符。按渠道性质灵活落地，但证伪与宽窄搭配的意图要保留。

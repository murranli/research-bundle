# 报告撰写准则（compose 阶段参考）

> 「成品报告」节点的前半——内容组织与逻辑，不含美化。只决定"说什么、怎么排"，不决定"长什么样"。

## 一 · 观点先行（修 v1.0"宽泛标题"问题）
每个章节标题必须是**被提炼出的一个核心观点**，让读者读标题就被吸引、知道这章的判断；正文再给论证。
**禁止**用宽泛名词性标题。正反例：
- ✗「综合分析结论」「权益类型与层次」「产品能力概述」「市场现状」
- ✓「问界靠后排屏赌的是家庭场景，理想三屏更偏驾驶位效率」「两车语音时延差<0.05s，体验差异不在快慢而在容错」
要求：一章一个核心观点；观点下挂其论证支撑；读者先被观点吸引，再读理由。

## 二 · 论证言之有物（修 v1.0"大词黑话"）
- 每个论证要点（point）落到具体事实/数字/机制，引用 evidence（`evidence_refs`）。
- 不用"赋能/抓手/闭环/生态位"这类空话堆砌；能用具体说法就不用大词。
- `fact` 与 `inference` 分明：推断要能回指支撑它的事实证据，并标 confidence。

## 三 · 不死磕单源、不为填满而堆砌（修 v1.0"缝隙都填满"）
- 把证据放进**最有效的小结单元**，不要把同一信源在多个章节反复滥用充数。
- 某观点若仅单源支撑：要么如实标低置信，要么并入"局限与待补"，**不要硬撑成一个看似完整的章节**。
- 信源不足的点 → 进 `limitations` **留白**，明说"待下一轮补充"，不臆造、不灌水。

## 四 · 紧扣原始诉求、不收缩（对应导向问题）
- 大纲回答的必须是 `original_request` / standard_statement，不是被悄悄缩小的版本。
- `scope.serves` 非空时：每章补 `design_implication`（或对应下游目的的启示），让全文服务于最终目的，而非泛泛综述。
- 收尾用 `scope_fidelity` 自证：这份大纲是否完整回应了原始诉求。

## 五 · 结论结构依课题性质自定义（不套死模板）
按 `report_type` 选组织骨架：
- 对比类 → 先给"差异全景/裁决"，再逐维度对比，最后给"对我方的取舍建议"。
- 评估类 → 按判据维度逐条评分/定性，给总判断 + 信心。
- 探索/认知类 → 按结构维度铺陈，先框架后细节。
- 决策类 → 选项 × 评价维度 → 推荐 + 理由 + 风险。
不强行套一个固定模板；让结构服务于课题。
